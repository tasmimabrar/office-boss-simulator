## Technical Architecture — Optimization Strategy

> Every system runs at the fidelity it deserves. Distance, visibility, and player relevance determine
> simulation budget. Nothing runs at full cost all the time.

### World Streaming
- [ ] The open-world city is divided into sectors (chunks)
- [ ] Only sectors within a configurable radius of the player are loaded and active
- [ ] Distant sectors exist as static impostor meshes only — no physics, no AI, no simulation
- [ ] Sector load/unload is async using Godot's `WorkerThreadPool` — no frame stutter on transitions
- [ ] Interior spaces (office floors) are separate scenes, loaded additively when the player enters

### Universal Simulation LOD — "The Three Tiers"
Every dynamic object (NPC, vehicle, physics body, effect) runs in one of three tiers:

| Tier | Distance | What runs |
|------|----------|-----------|
| **Full** | < 40m | Full AI, full physics, full animation, full effects |
| **Medium** | 40–120m | Simplified AI (no pathfinding recalc every frame), interpolated animation, no detailed physics |
| **Distant** | > 120m | Fake movement only — pre-baked loop animation, no physics, no AI |

- [ ] Every NPC and vehicle has a `SimulationLODComponent` that monitors distance to player and transitions between tiers
- [ ] Tier transitions are smoothed — no visible pop
- [ ] Tier thresholds are tunable per object type (boss NPCs stay Full longer; street crowd Distant sooner)

### NPC & AI Budget
- [ ] Hard cap on concurrent full-tier AI agents: **~60 NPCs** running pathfinding simultaneously
- [ ] Beyond that cap, agents queue and time-slice — pathfinding updates spread across multiple frames
- [ ] Schedule simulation (where NPCs *should* be) runs even for distant/unloaded NPCs — they teleport to the correct position when the player approaches
- [ ] Syndicate tactical AI runs on `WorkerThreadPool` — decision trees computed off the main thread
- [ ] Vision/hearing raycasts batched into groups, spread across frames — not every NPC checks every frame

### Physics Budget
- [ ] Jolt Physics sleeping — all `RigidBody3D` objects sleep when velocity drops below threshold (automatic with Jolt)
- [ ] `process_mode = DISABLED` on physics bodies in unloaded sectors — zero cost when not visible
- [ ] Ragdolls: max **8 simultaneous active ragdolls**. Beyond that, oldest ragdoll freezes in place
- [ ] Debris/fragments: capped at **32 active fragments** per destruction event via object pool
- [ ] Vehicle physics: only `VehicleBody3D` simulation within Full tier. Medium/Distant vehicles use kinematic path following

### Rendering Budget
- [ ] All `MeshInstance3D` nodes use Godot's built-in visibility ranges + `VISIBILITY_RANGE_FADE_SELF`
- [ ] Three LOD meshes per significant object (high/mid/low poly)
- [ ] GPU particles instead of CPU particles everywhere possible
- [ ] Compute-based fluid sim only spawns and runs for the duration of a setpiece event — destroyed afterward
- [ ] VoxelGI baked per sector — never dynamic. SSAO always on. SSR only on tagged surfaces (floors, glass, screens)
- [ ] Occlusion culling enabled on all levels — Godot's built-in occluder system

### Threading Model
- [ ] **Main thread:** rendering, physics, input, core game logic
- [ ] **WorkerThreadPool jobs:** NPC AI decision trees, pathfinding requests, world sector loading, save/load I/O
- [ ] **GPU compute:** fluid simulation, large particle sorting, any deformation shaders
- [ ] GDScript `await` + signals for all async operations — never block the main thread with a busy loop

### Memory Strategy
- [ ] Object pools for: NPCs, vehicles, debris fragments, projectiles, particle emitter instances
- [ ] Audio: `AudioStreamPlayer3D` pool — fixed count, reassigned by priority when exhausted
- [ ] No `instantiate()` in any hot path — all frequently spawned objects come from pools
- [ ] Streaming textures: use Godot's `StreamTexture2D` for large textures not always in view

### Profiling & Debug Contracts
- [ ] Every new system ships with a budget constant (e.g. `const MAX_ACTIVE_NPCS: int = 60`) — not magic numbers
- [ ] Debug overlay (toggle with F3 in dev builds) shows: active agent count, active physics bodies, current sector, frame time per system
- [ ] Performance regression test: game must maintain 60fps on the target minimum spec before any feature is merged

## Physics Systems (In Scope)

### Fluid / Smoke Simulation
- [ ] GPU particle-based smoke and fire for everyday events (server fire, HVAC vents, breakroom microwave)
  - Implemented via `GPUParticles3D` + custom `.gdshader` with 3D curl noise and dissipation
  - Cost: cheap, looks convincing in-context
- [ ] Real compute-based fluid sim for significant setpiece moments (major fire, gas leak, explosion)
  - Implemented as a 3D Navier-Stokes solver in GLSL compute shaders via Godot's `RenderingDevice`
  - 64³ grid resolution target for real-time performance on modern hardware
  - Scoped to moments where it matters — not running constantly

### Ragdolls
- [ ] Enemies, employees, and NPCs transition to ragdoll physics on death or knockback
  - Implemented via `PhysicalBoneSimulator3D` attached to character skeletons
  - Jolt Physics handles bone collider simulation
  - Blend between animation-driven and physics-driven per-bone at runtime
  - Security encounters, syndicate hits, workplace "accidents" all use this

### Vehicles & Deformation
- [ ] Driveable vehicles in the open world (player car, syndicate vehicles, delivery trucks, buses, vans)
  - Implemented via `VehicleBody3D` with Jolt's vehicle controller
- [ ] Multiple vehicle classes: sedans, SUVs, vans (syndicate), delivery trucks, service vehicles
- [ ] Vehicle AI — parked cars, traffic flow, syndicate pursuit vehicles, drive-bys
  - AI drivers use `NavigationAgent3D` on road navmesh lanes
- [ ] Vehicle collision deformation
  - Primary: pre-baked damage states (swap meshes at damage thresholds) — ships with the base game
  - Stretch: runtime vertex deformation via compute shader — defer to post-launch if time allows
- [ ] Vehicles can be destroyed — transitions to physics debris chunks (`RigidBody3D` fragments)
- [ ] Player can enter/exit any vehicle (carjacking included)
- [ ] Vehicle inventory — trunk for transporting goods, contraband, equipment
- [ ] Road traffic simulation — background vehicles on a spline/navmesh circuit to make the city feel alive

---

### NPC Pathing & AI Navigation
- [ ] All NPCs (employees, pedestrians, syndicate, security) navigate using Godot's `NavigationAgent3D`
- [ ] Navigation mesh baked per level — office floors and open-world city are separate navmeshes
- [ ] Office NPCs follow daily schedules:
  - [ ] Arrive → walk to desk → work → break (kitchen/bathroom) → back to desk → leave
  - [ ] Schedule deviates based on mood, events, and player actions
- [ ] Crowd simulation for busy areas (street pedestrians, building lobbies)
  - Use `NavigationObstacle3D` for dynamic avoidance between NPCs
- [ ] Syndicate NPCs have tactical pathfinding: take cover, flank, retreat
- [ ] NPCs react to environmental hazards: fire, smoke, explosions — flee via navigation
- [ ] NPCs can use vehicles — drive to work, flee in cars, syndicate ambushes arrive by vehicle
- [ ] Security guards patrol on navmesh with configurable patrol routes
- [ ] NPC line-of-sight and awareness system (not just nav — they need to *notice* things)
  - Raycast-based vision cone; sound propagation for alerts
- [ ] Crowd density affects navmesh performance — cap active pathfinding agents, use LOD for distant NPCs
  - Distant NPCs run simplified "fake walk" animations without full nav computation
